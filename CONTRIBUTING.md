# CONTRIBUTING

## Как писать новую документацию

Перед созданием документа определите цель, аудиторию, статус и раздел размещения.

Каждый документ должен использовать `DOCUMENT_TEMPLATE.md`, если для него не предусмотрен отдельный шаблон.

Документ не должен содержать неподтвержденные продуктовые, технические или бизнес-решения.

## Правила форматирования

- Используйте Markdown.
- Пишите кратко и конкретно.
- Один документ должен решать одну задачу.
- Не используйте декоративное оформление без необходимости.
- Не смешивайте утвержденные решения и черновые заметки.

## Правила Markdown

- Заголовок первого уровня должен быть один.
- Уровни заголовков не должны пропускаться.
- Таблицы используются для структурированных данных.
- Списки используются для перечислений, а не для длинных объяснений.
- Ссылки должны быть относительными внутри репозитория.

## Правила именования

Формат имени файла:

```text
NNNN_Document_Name_tX.md
```

Где:

- `NNNN` - порядковый номер документа в разделе.
- `Document_Name` - краткое название на английском в `Pascal_Case` или `Title_Case` через подчеркивания.
- `tX` - версия документа.

Пример:

```text
0001_Project_Mission_t1.md
```

## RFC

RFC используется для предложения изменений, которые требуют обсуждения до принятия.

Правила:

- RFC создается в `docs/15_RFC`.
- RFC должен использовать `RFC_TEMPLATE.md`.
- RFC начинается со статуса `Draft`.
- После обсуждения RFC получает статус `Approved`, `Deprecated` или остается в `Review`.
- Утвержденный RFC может привести к созданию ADR.

## ADR

ADR используется для фиксации принятого решения.

Правила:

- ADR создается в `docs/16_ADR`.
- ADR должен использовать `ADR_TEMPLATE.md`.
- ADR фиксирует решение, контекст и последствия.
- ADR не заменяет полноценную спецификацию.
- Устаревший ADR получает статус `Deprecated`, но не удаляется.
